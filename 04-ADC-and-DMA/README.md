# 04 — UART Transmit via DMA

Transmits a fixed string over **UART2 using DMA** instead of a blocking `HAL_UART_Transmit()` call, freeing the CPU during the transfer and using a registered DMA completion callback to signal when transmission finishes.

## What It Does

- USART2 (PA2/PA3 — the ST-Link virtual COM port on the Nucleo board) is configured for 115200 baud, 8N1
- A DMA channel (`hdma_usart2_tx`) is linked to USART2's TX in `HAL_UART_MspInit()` (auto-generated by CubeMX once a DMA request is added to USART2_TX in the `.ioc`)
- A custom completion callback is registered manually via `HAL_DMA_RegisterCallback()`, rather than relying on `HAL_UART_Transmit_DMA()`'s built-in handling — this project drives the DMA transfer and UART register bits directly for a closer look at what the HAL abstraction normally does under the hood
- Every second, the main loop enables the UART's DMA transmit request bit and kicks off a DMA transfer directly into the UART's Transmit Data Register (TDR):

```c
huart2.Instance->CR3 = USART_CR3_DMAT;
HAL_DMA_Start_IT(&hdma_usart2_tx, (uint32_t)msg,
                 (uint32_t)&huart2.Instance->TDR, strlen(msg));
```

- When the transfer completes, the registered callback disables the DMA transmit request bit and toggles the onboard LED (PA5) as a visual "transfer done" indicator:

```c
void DMATransferComplete(DMA_HandleTypeDef *hdma) {
    huart2.Instance->CR3 &= ~USART_CR3_DMAT;
    HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);
}
```

## Peripherals / Concepts

- **UART transmission via DMA** — moving data to the UART's TDR register without CPU involvement per-byte
- **Direct register manipulation** — manually setting `USART_CR3_DMAT` and starting the DMA transfer, instead of using the higher-level `HAL_UART_Transmit_DMA()` wrapper, to better understand what that wrapper does internally
- **DMA callback registration** — `HAL_DMA_RegisterCallback()` to hook a custom function into the DMA transfer-complete event instead of relying on default HAL behavior
- Continued use of USART2 → ST-Link VCP for output, verified over PuTTY at 115200 baud on the assigned COM port

## Hardware

- NUCLEO-G474RE
- No external components required — output is viewable over the onboard ST-Link virtual COM port

## Debugging Notes

Earlier versions of this project called `BSP_COM_Init(COM1, ...)` in addition to the manual `MX_USART2_UART_Init()` + DMA setup shown above. On the NUCLEO-G474RE, `COM1` in the BSP maps to the same physical peripheral (USART2) already being configured manually. Calling both meant the BSP's `HAL_UART_Init()` ran a second time on the same peripheral right before the main loop, which triggered `HAL_UART_MspDeInit()`/re-init internally and disrupted the DMA channel that had just been linked to `huart2` — resulting in complete silence on the serial terminal despite the code appearing to run correctly.

**Diagnosis process:**
1. Verified DMA handle was being properly initialized in `HAL_UART_MspInit()` (it was — ruled out an uninitialized DMA handle)
2. Isolated the problem by temporarily swapping the DMA transfer for a plain blocking `HAL_UART_Transmit()` call — if basic UART worked but DMA didn't, the problem was DMA-specific; if nothing worked at all, the conflict was happening at the peripheral level
3. Traced it to the redundant `BSP_COM_Init()` call fighting over the same UART instance as the manually configured DMA-based transmit path

**Fix:** removed `BSP_COM_Init()` entirely from this project, since UART is being driven by hand here rather than through the BSP's COM abstraction. Going forward, when starting a new CubeMX project intended for manual UART/DMA configuration, the BSP COM component is disabled at the `.ioc` level (via the Bsp panel in Pinout & Configuration) rather than removed after the fact, to avoid this conflict recurring on regeneration.
